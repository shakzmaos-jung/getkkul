# CLAUDE.md — getkkul (겟꿀)

> 이 파일은 Claude Code 에이전트가 이 프로젝트에서 일하는 방식을 규정한다.
> 스펙의 단일 진실 공급원은 `docs/PRD.md`(v0.4)와 `docs/SSR.md`(v0.2)이다. 작업 전 반드시 읽는다.

## 프로젝트 컨텍스트

- **무엇**: 구독한 유튜브 채널을 대신 감시해, 새 영상의 핵심을 요약하고 하루 3회 정시에 밀어주는 개인용 큐레이션 서비스.
- **누구를 위해**: 나(운영자) + 소수의 친구(신뢰 그룹). 일반 공개 아님.
- **성격**: 사업 진입이 아니라 (1) AI 에이전틱 개발 파이프라인 학습, (2) 나에게 최적화된 경량 버전. 차별화를 과장하지 않는다.

## 개발 방식 (SDD → TDD)

- 모든 작업은 스펙에서 출발한다. `docs/PRD.md` → `docs/SSR.md`(요구사항 REQ + 수용 기준 AC)가 유일한 근거다.
- 구현은 SSR의 수용 기준(AC)을 **실패 테스트로 먼저 작성(Red)** → 통과 코드(Green) → 리팩터 순으로 진행한다.
- 사용자(나)는 테스트 코드를 쓰지 않는다. 테스트는 AC에서 파생한다. 각 AC는 최소 1개의 테스트로 매핑되어야 한다.
- 스펙에 없는 기능을 임의로 추가하지 않는다. 필요하면 먼저 스펙 변경을 제안한다.

## 사람 vs 에이전트 경계 (Human-on-the-loop)

- 사용자(나)는 상류 결정만 담당한다: 스펙 승인, 구현 계획 승인, 최종 머지.
- 그 외 구현 디테일은 검증 게이트(테스트·린트·타입·CI) 통과를 조건으로 자율 진행한다.
- 나는 매 단계 승인이 아니라, 중대 결정과 주요 변경 보고만 처리한다.

## 에스컬레이션 정책 (진행 전 반드시 정지하고 확인)

- 데이터 스키마 변경(테이블·제약·인덱스 변경)
- 외부 의존성 추가(새 라이브러리, 새 서드파티 서비스)
- 비가역적 삭제 또는 파괴적 작업
- 아키텍처 방향 전환(특히 아래 "격리 경계" 변경)
- 스펙(PRD/SSR) 해석이 모호하거나 서로 충돌할 때
- 유튜브 콘텐츠 획득의 법적·약관 회색지대와 관련된 방식 선택
- 비용에 크게 영향을 주는 결정(예: 전 영상 강제 오디오 전사)
→ 위 항목은 진행 전 정지·질의한다. 내가 내린 결정은 이 파일 또는 `docs/adr/`에 기록한 뒤 재개한다(같은 질문 재발 방지).

## 아키텍처 & 격리 원칙 (핵심)

두 경계를 **단일 인터페이스 뒤로 격리**한다. 이 경계 밖 코드는 구현 세부를 몰라야 한다.

1. **콘텐츠 획득**: `fetchContent(source)` — RSS 감지 + 자막 우선/오디오 STT 폴백. 유튜브 방어 변화·소스 추가(v2 네이버 등)에 대응해 교체 가능해야 한다.
2. **알림 발송**: `notify(user, message)` — v1은 이메일(Resend). 이후 텔레그램·카카오 구현체로 교체 시 상위 코드가 바뀌지 않아야 한다.

- **감지·처리와 발송을 분리**한다: 전사·요약은 영상 감지 시점에 미리 수행·저장하고, 발송 시각에는 준비된 것만 밀어낸다.

## 기술 스택 & 규약

- **DB·인증**: Supabase (Auth: Google 소셜 로그인 / RLS / Postgres)
- **호스팅·프론트**: Vercel (v1은 `getkkul.vercel.app` 서브도메인). 프론트 프레임워크는 계획 단계에서 확정(Vercel 친화적인 Next.js 권장).
- **스케줄**: Vercel Cron 또는 Supabase pg_cron/Edge Functions. 발송 KST 07:30 / 11:30 / 17:30.
- **이메일**: Resend.
- **요약**: LLM API(프로바이더는 계획 단계에서 확정).
- **전사**: 자막 우선, 실패 시 오디오 STT(Whisper류; 프로바이더는 계획 단계에서 확정).
- **타임존**: 모든 타임스탬프 UTC 저장, 스케줄·표시는 KST 변환.
- **언어**: 요약 한국어 기본 + 웹에서 영어 전환. 요약 캐시 키는 (video_id, length_mode, language).

## 검증 하네스

- 모든 변경은 테스트·린트·타입체크를 통과해야 한다. 게이트 통과가 완료 기준이다.
- 실패 시 사람을 부르지 말고 스스로 교정한다(self-correct).
- 개별 영상의 획득/요약/발송 실패가 전체 잡을 중단시키지 않는다(격리·회복력).
- 반복되는 실수·교훈은 이 파일에 적어 다음 세션에 전파한다.

## 데이터 & 보안 규칙

- 사용자 스코프 테이블(profiles, user_settings, subscriptions, deliveries)에 RLS 적용: `user_id = auth.uid()` 행만 접근.
- 재발송 방지(멱등성)는 DB unique 제약으로 강제한다: `deliveries UNIQUE(user_id, video_id)`. 앱 로직에만 의존하지 않는다.
- 시크릿·API 키는 환경변수로만 관리. 서비스 롤 키는 서버측에서만 사용. 코드·로그·대화에 평문 노출 금지.
- 개인정보 최소 수집(이메일·구독 소스). 계정 삭제 시 cascade.

## 버전 관리 (ADR-0012)

- SemVer(`MAJOR.MINOR.PATCH`), 단일 소스는 `package.json.version`(→ 푸터 자동 표시). 사용자 대면 변경이 있는 릴리스는 같은 PR 에서 버전 bump(PATCH=버그·문구 / MINOR=기능 / MAJOR=파괴적) + `CHANGELOG.md` 항목 추가, 머지 후 `git tag -a vX.Y.Z && git push --tags`.
