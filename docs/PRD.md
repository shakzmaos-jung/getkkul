# PRD — getkkul (겟꿀)

**버전** v0.4 · **작성자** Chess · **상태** v1 스코프 확정 · **최종 수정** 2026-07-04
**코드네임** getkkul(겟꿀) — "좋은 것(꿀)을 얻는(get) 서비스". 이름만으로 정체가 연상되고 톤이 가볍고 친근함.

---

## 1. 한 줄 요약

내가 구독하는 유튜브 채널을 대신 감시해, 새 콘텐츠의 핵심만 정해진 시각에 밀어주는(push) 개인용 큐레이션 서비스 **getkkul**. 소수의 신뢰 그룹(나 + 친구)이 각자 Google 로그인으로 접속해 자기 채널을 구독하고 자기 다이제스트를 받는다. 알림 채널은 이메일로 시작해 텔레그램·카카오톡으로 확장한다.

## 2. 배경 & 문제 정의

관심 콘텐츠는 늘어나는데 다 볼 시간은 없다. 기존 요약 툴 상당수는 "URL을 붙여넣으면 요약해주는" 온디맨드(pull) 방식이라, "무엇을 볼지 고르는 노동"과 "요약을 돌리는 노동"이 사람에게 남는다. getkkul은 그 노동을 뒤집는다. 관심 채널을 한 번 구독해두면, 이후엔 시스템이 새 콘텐츠를 감지·요약해 정해진 시각에 밀어준다. 가치의 핵심은 요약 자체가 아니라 "나 대신 감시하고 정해진 리듬으로 떠먹여주는 큐레이션"이다.

## 3. 포지셔닝 & 시장 현실 (정직한 기준선)

이 컨셉은 새롭지 않다. 국내 Lilys가 이미 구독 → 자동 요약 → 이메일·앱 알림, 심지어 카카오톡 채널까지 성숙하게 구현했고, 해외에도 NotebookLM·Recall 등 강자가 있다. 따라서 본 프로젝트는 **신규 사업 진입이 아니라, (1) AI 에이전틱 개발 파이프라인을 직접 굴려 체화하기 위한 개인 학습 프로젝트이자, (2) 나와 소수 친구에게 최적화된 경량 버전**으로 포지셔닝한다. 차별화를 과장하지 않는다. 성공 기준은 시장 점유가 아니라 "내가 매일 실제로 쓰는가 + 파이프라인을 완주했는가"다.

## 4. 목표 (Goals)

- 구독한 유튜브 채널의 신규 영상을 자동 감지하고, 핵심을 요약해 정해진 시각에 알림 채널로 발송한다.
- 소수 신뢰 사용자(나 + 친구)가 각자 Google 로그인으로 접속해 자기 채널 구독을 관리하고 자기 다이제스트만 받는다.
- 한국어와 영어권(영미) 콘텐츠를 함께 다룬다(영어권에 유익한 콘텐츠가 많으므로). 요약은 한국어를 기본으로 하되 영어 전환을 제공한다.
- 알림 채널을 교체 가능하게 설계해 이메일 → 텔레그램 → 카카오톡 순으로 확장한다.

## 5. 비목표 (Non-goals) — v1에서 하지 않는 것

- 유튜브 외 소스(네이버 블로그 등) → v2
- 채널 구독 외의 소스 단위(개별 영상 URL, 재생목록) → v2 이후
- 비공개·유료 프리미엄 콘텐츠 접근 → 보류
- 영상+시각 결합 이해(스마트 샘플링·키프레임 캡처) → v2
- 의미 기반 중복 제거(같은 뉴스 다중 소스 병합) → v2 이후 (단, "이미 발송한 콘텐츠 재발송 방지"는 v1 필수)
- 결제·구독 과금 → v3
- 운영자 어드민 대시보드 → v3
- 실시간 즉시 알림(이벤트 기반) → 정시 발송으로 대체
- 일반 대중 공개 런칭 → 신뢰 그룹 한정

## 6. 대상 사용자

1차 사용자는 나 자신과 사용을 권할 소수의 친구들. "단일 사용자"가 아니라 "소수의 신뢰 그룹"이며, 이 때문에 사용자별 데이터 격리와 최소 계정 체계가 v1부터 필요하다.

## 7. 핵심 기능 (v1, 하이레벨)

- **인증 / 계정**: Google 소셜 로그인(Supabase Auth). 사용자 데이터는 RLS로 `user_id` 기준 격리 — 각자 자기 구독·다이제스트만 접근. (카카오톡 알림 단계에서 별도의 카카오 계정 연결을 추가: "Google = 신원, 카카오 = 배송"의 2단 구조.)
- **소스 등록(사용자 웹)**: 관심 유튜브 **채널을 구독** 등록·삭제하는 최소 CRUD 화면. v1은 채널 단위만 지원(개별 영상 URL·재생목록 제외).
- **콘텐츠 획득**: 구독 채널의 신규 영상을 **RSS로 감지** → 자막이 있으면 우선 사용, 없거나 막히면 오디오를 받아 STT(Whisper류)로 전사하는 폴백. 이 레이어는 "이 소스의 콘텐츠를 가져와라"라는 단일 인터페이스 뒤로 격리한다.
- **요약**: 전사 텍스트를 LLM으로 핵심 요약. **사용자가 요약 길이를 선택**(짧게 / 보통 / 길게). 의미 기반 중복 제거는 하지 않으나, **처리·발송 이력을 추적**해 같은 영상이 반복 발송되지 않게 한다(idempotency).
- **알림 발송**: 사용자별 정시(하루 3회)에 다이제스트를 발송. 발송 채널은 교체 가능한 인터페이스 뒤에 두고 v1은 이메일로 구현.

## 8. 알림 채널 로드맵

발송 로직은 "이 사용자에게 이 메시지를 보내라"라는 단일 인터페이스로 추상화하고 구현체만 교체한다.

| 단계 | 채널 | 유의점 |
|---|---|---|
| v1 | 이메일 | 가장 단순한 배관. 파이프라인 검증에 최적. |
| v1.x | 텔레그램 봇 | 개인 자동화에 가장 간단(OAuth·검수 불필요). |
| v2 | 카카오톡 "나와의 챗" | 사용자별 카카오 OAuth(talk_message) + 리프레시 토큰 관리 필요. 각자 자기 챗 연결이 "친구에게 대신 보내기"보다 마찰이 적음. 사업자 방송용 알림톡은 v3. |

## 9. 발송 스케줄 & 처리 분리

- **발송 시각(KST 고정)**: 하루 3회 — 출근 07:30 / 점심 11:30 / 퇴근 17:30.
- **구현**: Vercel Cron 또는 Supabase pg_cron/Edge Functions로 KST 세 시각 스케줄.
- **감지·처리와 발송 분리**: 새 영상 감지 시점에 전사·요약을 미리 수행·저장하고, 발송 시각에는 준비된 요약만 밀어낸다(발송마다 재처리 금지).
- **각 발송 내용**: 직전 발송 이후 새로 준비된 항목을 담는다. **새 항목이 없으면 "새 소식 없음"을 짧게 발송한다**(항상 발송 — 파이프라인이 살아있다는 하트비트 역할 겸함).

## 10. 범위 & 단계 (로드맵)

- **v1 (지금)**: 유튜브 단독 · 공개 콘텐츠 · 채널 구독 · 자막/오디오 전사 기반 요약(길이 선택) · 하루 3회 정시 발송(빈 슬롯은 "새 소식 없음") · Google 로그인 기반 소수 사용자 · 이메일 알림 · 결제·어드민·의미 dedup 없음.
- **v1.x**: 알림을 텔레그램으로 확장.
- **v2**: 카카오톡 알림, 네이버 등 소스 확장, 개별 URL·재생목록 소스 단위, 영상+시각 스마트 샘플링, 유튜브 구독 자동 임포트(OAuth).
- **v3 (서비스화 가정)**: 구독 과금, 운영자 어드민 대시보드, 카카오 알림톡(사업자), 일반 공개, 콘텐츠 접근 합법 경로(공식 API·라이선스·제휴).

## 11. 성공 지표

(1) 내가 매일 다이제스트를 여는가(주간 열람 일수), (2) 요약으로 직접 시청을 갈음한 비율(체감 포함), (3) 초대한 친구의 1주 리텐션, (4) 스펙 승인부터 배포까지 사람 개입 최소로 파이프라인을 한 바퀴 돌렸는가. 정량 목표는 v1 가동 후 실측·보정.

## 12. 기술 접근 & 아키텍처 개요

- **소스 관리**: GitHub
- **DB · 인증**: Supabase (Auth + RLS + Postgres)
- **스케줄 잡**: Supabase pg_cron/Edge Functions 또는 Vercel Cron (KST)
- **호스팅 · 프론트**: Vercel (v1 배포는 `getkkul.vercel.app` 서브도메인으로 충분; 커스텀 도메인은 후속)
- **요약**: LLM API
- **핵심 설계 원칙 (격리)**: 콘텐츠 획득 레이어와 알림 발송 레이어를 각각 단일 인터페이스 뒤로 격리한다. 유튜브 방어 변화·소스 추가·알림 채널 교체를 나머지 파이프라인을 건드리지 않고 대응하며, SDD/TDD에서 잘 정의된 경계 모듈로 다루기 위함.

## 13. 제약 & 가정 (법적·기술적)

- 유튜브 자막·오디오 획득은 기술적으로 가능하나 약관상 회색지대. 개인·소규모 신뢰 그룹 범위에서 운영하며, 서비스화(v3) 시 공식 API·라이선스·제휴 재검토.
- 유튜브 Data API v3는 자막을 제공하지 않으므로 사용자별 유튜브 API 키 등록은 v1에 불필요. 신규 감지는 채널 RSS로 처리.
- 비공식 획득 경로는 깨질 수 있으므로 획득 레이어를 교체·복구 가능하게 격리.
- 오디오 전사는 자막보다 안정적이나 비용이 든다. 자막 우선, 오디오 폴백으로 비용 관리.
- 오디오·텍스트만으로는 시각 의존 콘텐츠를 놓칠 수 있다. v1은 한계로 명시, v2 보강.
- 사용자 데이터(이메일, 구독 소스)를 다루므로 v1부터 데이터 최소 수집·삭제 가능성 습관화.
- 카카오톡 무인 발송은 토큰 만료 대응(리프레시) 필수.

## 14. v1 확정 사항 (닫힌 결정)

- **코드네임**: getkkul(겟꿀). 도메인 TLD는 후속(`.com`은 선점됨, `.io/.app/.co/.net/.me` 가용 확인). v1은 Vercel 서브도메인 사용.
- **소스 등록 단위**: 채널 구독만.
- **요약 깊이**: 사용자 선택(짧게 / 보통 / 길게). 각 모드의 구체 정의는 SSR에서 확정.
- **빈 발송 처리**: "새 소식 없음"을 짧게 발송(항상 발송).
- **알림 채널**: 이메일 → 텔레그램 → 카카오톡(나와의 챗).
- **발송 주기**: 하루 3회(07:30 / 11:30 / 17:30 KST).
- **인증**: Google 로그인 + RLS.
- **중복 처리**: 의미 dedup 제외, 발송 이력 추적 유지.
- **언어**: 한국어·영어권 콘텐츠 모두 다룸. 요약은 한국어 기본 + 영어 전환 제공.
