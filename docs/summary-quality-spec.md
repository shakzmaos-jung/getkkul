# SSR 부록 — getkkul 다이제스트 요약 품질 개선

**버전** v0.1 · **작성자** Chess (스펙 초안: 협업) · **상태** v1 스펙 확정 · **기반** getkkul SSR v0.2, 현재 `lib/summary/summarize.ts`·`format.ts`, videos/summaries/content_terms 스키마
**용도** 각 AC는 TDD 테스트의 원천이다. 판정 가능하게 기술한다.
**표기** REQ = 요구사항, AC = 수용 기준. 💰 = 비용 영향.

---

## 0. 배경 & 확정 결정

현재 요약은 (1) 길이 역전(long < normal < short 발생), (2) 문장 수 기준이라 깊이가 안 늘어남, (3) 전사 용어 오류(S&P500→SMP 500) 전파, (4) 콘텐츠 빈약도와 무관하게 항상 3종 생성, (5) long 시스템 프롬프트가 얇음의 문제가 있다.

**확정 방향**
- 길이 → **정보 계층(깊이)** 재정의. 문장 수 검증을 단조성 검증으로 교체·보강.
- long은 **핵심 사실 요약 / 맥락·시사점·인사이트를 단락 구분**해 제시(Chess 확정).
- 콘텐츠 빈약 시 무리하게 long 생성하지 않고 그 사실을 밝힌다(적응형 깊이).
- 전사 용어를 **보수적으로** 교정(과교정 금지).
- long 핵심 문장 하이라이트.
- **피드백 루프**: 콘텐츠 카드 하단 좋아요/별로예요 평가를 쌓아 시스템 프롬프트 개선에 활용(Chess 확정).
- 확장: 용어 설명 · 타임스탬프 · on-demand "더 자세히/더 쉽게"(AI질의와 별개의 자동 제공 가치).
- long에 한해 `reasoning_effort='low'`(💰 소폭). Batch API 미사용(실시간 우선). 오디오 STT는 당분간 유료 API 유지(💰 관측 대상).

## A. 정보 계층 재정의 (핵심)

**REQ-A1** 각 길이 모드를 문장 수가 아니라 정보 계층으로 정의한다.
- AC-A1.1 short = TL;DR(무엇을 다뤘나) 핵심 한두 요점.
- AC-A1.2 normal = 핵심 요점 + 왜 중요한지 맥락.
- AC-A1.3 long = **두 단락 구분 구조**: ① 핵심 사실 요약(전사에서 뽑은 사실·수치·사례), ② 그로부터 도출한 맥락·시사점·인사이트. 두 파트를 시각적으로 구분해 저장·표시한다.
- AC-A1.4 원본에 없는 내용은 생성하지 않는다(인사이트도 사실에 근거).

**REQ-A2** long 시스템 프롬프트를 정보 계층·단락 구분·근거 준수·교정·하이라이트를 지시하도록 재작성한다.
- AC-A2.1 프롬프트는 "몇 문장"이 아니라 "무엇을·어떤 층위로" 담을지 지시한다.
- AC-A2.2 프롬프트 문안은 버전 관리하고, 변경 시 회귀 테스트로 품질 확인(단조성·단락 구분·하이라이트 존재).

## B. 단조성 보장 (길이 역전 제거)

**REQ-B1** short ≤ normal ≤ long이 항상 성립한다.
- AC-B1.1 검증을 기존 "범위 단독 판정"에서 **단조성 판정**(정보량·길이·포함관계)으로 교체·보강한다.
- AC-B1.2 단조성 위반 시 재생성(교정)한다. 최종 실패 시에도 역전이 사용자에게 노출되지 않도록 방어(예: 상위가 하위보다 짧으면 보정).
- AC-B1.3 방식은 단일 호출 + 단조성 검증을 기본으로 하되(💰 거의 무변), 품질 미달 시 long-우선 생성 후 압축 파생으로 상향 가능(💰 소폭 증가) — 초기엔 전자.

## C. 적응형 깊이 (빈약 콘텐츠)

**REQ-C1** 콘텐츠 정보 밀도에 따라 제공 모드를 조절한다.
- AC-C1.1 모델이 요약과 함께 콘텐츠 깊이(depth ceiling)를 판정한다(같은 호출, 💰 무변).
- AC-C1.2 얕은 콘텐츠는 long을 무리하게 생성하지 않고, "이 영상은 내용이 짧아 '짧게'만 제공합니다" 류의 피드백을 표시한다.
- AC-C1.3 이때 상위 모드는 "제공 안 함" 상태로 저장/표시한다(빈 long 억지 생성 금지).

## D. 전사 용어 보수적 교정

**REQ-D1** 명백한 용어 오류만 보수적으로 교정한다.
- AC-D1.1 채널 도메인 힌트(구독 채널 맥락)를 프롬프트에 제공해 도메인 용어 교정을 돕는다.
- AC-D1.2 잘 알려진 고유명사·전문용어의 명백한 오인식(S&P500 등)만 교정한다.
- AC-D1.3 **과교정 금지**: 추측·생성으로 없던 개체를 만들지 않는다(예: 발음 유사로 엉뚱한 고유명사 삽입 금지). 확실치 않으면 원문 유지.
- AC-D1.4 (선택) content_terms 를 용어집으로 활용해 채널별 반복 용어 교정 정확도를 높인다.

## E. 핵심 문장 하이라이트

**REQ-E1** long에서 핵심 문장을 표시한다.
- AC-E1.1 스키마에 핵심 문장 마킹을 추가해, long에서 중요 문장을 강조(underline 등)로 렌더한다.
- AC-E1.2 하이라이트는 사실 파트·인사이트 파트 모두에서 가능.

## F. 피드백 루프 (평가 데이터 축적)

**REQ-F1** 사용자가 다이제스트를 간단 평가한다.
- AC-F1.1 콘텐츠 카드 하단에 좋아요(👍)/별로예요(👎)를 둔다. 1탭으로 평가, 재탭으로 취소·변경.
- AC-F1.2 평가는 (user_id, video_id, length_mode, language) 맥락과 함께 저장한다.
- AC-F1.3 평가는 사용자당 콘텐츠·모드별 1회(변경 가능). 익명 집계 가능한 형태.

**REQ-F2** 평가 데이터를 프롬프트 개선에 활용한다.
- AC-F2.1 평가를 집계해 저평가가 몰리는 패턴(모드·채널·길이 등)을 관측할 수 있는 지표를 만든다.
- AC-F2.2 이 지표를 근거로 시스템 프롬프트를 반복 개선한다(운영 루프). 프롬프트 버전과 평가 지표를 연결해 개선 전후를 비교한다.
- AC-F2.3 (초기) 자동 재학습이 아니라, 지표 → 사람이 프롬프트 개정 → 회귀 테스트의 HOTL 루프로 시작한다.

## G. 확장 (자동 제공 가치)

**REQ-G1** 용어 설명.
- AC-G1.1 어려운 핵심 용어에 짧은 설명(툴팁/각주)을 제공한다(content_terms 활용). 💰 같은 호출 내 처리 시 무변, 별도 호출 시 소폭.

**REQ-G2** 타임스탬프.
- AC-G2.1 핵심 문장이 원본 영상의 어느 시점인지 표시한다(전사에 타임코드가 있을 때). 없으면 생략.

**REQ-G3** on-demand 깊이 조절.
- AC-G3.1 콘텐츠 카드에서 "더 자세히 / 더 쉽게"를 그 자리에서 요청할 수 있다(고정 3티어와 별개, Lilys식). 💰 요청 시마다 호출 발생 → 결과 캐시로 반복 비용 억제. AI질의와 기능이 겹치지 않게 역할 구분(질의=사용자 질문, on-demand=깊이 재구성).

## H. 데이터 모델 (추가)

- **summaries**(활용): long의 사실/인사이트 단락 구분과 하이라이트를 `body`(jsonb)에 구조화 저장(예: {facts:[...], insights:[...], highlights:[...]}). depth 판정·"제공 안 함" 상태도 저장.
- **content_feedback**(신설): id, user_id, video_id, length_mode, language, rating enum['up','down'], created_at, UNIQUE(user_id, video_id, length_mode, language).
- **content_terms**(기존 활용): 용어 교정·설명 소스.
- (선택) **prompt_versions**: 시스템 프롬프트 버전과 배포 시각 기록(F2 개선 전후 비교용).

## I. 비용 영향 요약 (💰)

- A·B(단일호출)·C·D·E·F: 기존 단일 호출의 프롬프트·스키마 확장 → 영상당 추가 비용 거의 0. C(빈약 시 long 생략)는 오히려 절감.
- long `reasoning_effort='low'`: 영상당 0.05→0.15센트 수준(무시 가능).
- on-demand(G3): 사용자 요청 시 호출 발생 → 캐시로 억제.
- **최대 비용 항목은 요약이 아니라 오디오 STT(유료 API, 7/11 기준 약 7%)** — 당분간 유지하되 비용 추이 관측, 임계 초과 시 로컬(Mac mini) 전환 검토.
- Batch API 미사용(실시간 조회 우선).

## J. 수용성 시나리오 & 통과 기준

- **S1 단조성**: 다양한 전사에서 short ≤ normal ≤ long 항상 성립(역전 0건).
- **S2 단락 구분**: long이 [핵심 사실]과 [맥락·시사점·인사이트] 두 파트로 구분 저장·표시된다.
- **S3 적응형**: 빈약 전사 → long "제공 안 함" + 안내. 풍부한 전사 → 3종 정상.
- **S4 교정**: 도메인 힌트로 명백한 용어 오류 교정. 불확실 용어는 원문 유지(과교정 0건 — 없던 개체 삽입 금지).
- **S5 하이라이트**: long 핵심 문장이 마킹되어 렌더된다.
- **S6 피드백**: 카드 하단 👍/👎 저장·변경, 맥락과 함께 기록, 집계 지표 산출.
- **S7 무회귀**: 기존 발송·조회·캐시(모드/언어별) 정상.
- 통과 기준: S1(역전 0)·S4(과교정 0)는 무결점. 나머지 전부 통과.

## K. 확정값 요약

정보-계층 재정의 / long 2단락(사실↔인사이트) / 단조성 검증 / 적응형 깊이(빈약 시 축소+안내) / 보수적 교정(과교정 금지) / long 하이라이트 / 👍👎 피드백→프롬프트 개선(HOTL) / 확장(용어설명·타임스탬프·on-demand) / long=low / Batch 미사용 / STT 유료 유지·관측.
