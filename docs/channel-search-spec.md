# SSR 부록 — getkkul 채널 검색 등록 (like 검색 → 선택 → 저장)

**버전** v0.1 · **작성자** Chess (스펙 초안: 협업) · **상태** v1 스펙 확정 · **기반** getkkul PRD v0.4 / SSR v0.2, 현재 코드베이스
**용도** 각 AC는 TDD 테스트의 원천이다. 판정 가능하게 기술한다.
**표기** REQ = 요구사항, AC = 수용 기준.

## 0. 배경 & 확정 결정

현재 채널 등록은 사용자가 핸들·URL을 직접 입력해야 해서, 유튜브 앱을 오가는 마찰이 크고 구독 채널 등록(=락인) 확대를 방해한다. 개선: getkkul 안에서 채널 제목을 검색(like)해 후보 목록에서 선택·저장한다.

**핵심 리스크**: YouTube Data API `search.list`는 호출당 100유닛(일반 호출의 100배)이며, getkkul은 이 쿼터(일 10,000유닛)를 이미 감지 폴백에 쓴다. 검색이 쿼터를 소진하면 **감지가 막혀 콘텐츠 누락**으로 번진다.

**확정 방식(하이브리드)**: 로컬 채널 카탈로그 우선 → 결과 캐시 → YouTube API 폴백(디바운스·최소 글자수·일일 상한) → 직접 입력(URL/핸들) 폴백. 검색으로 확정된 채널은 카탈로그에 축적해 다음부터 무료화한다. (별도 API 키 물리 격리는 향후 확장.)

## A. 검색 UX

**REQ-A1** 사용자는 채널 제목을 입력해 후보 목록에서 선택·저장한다.
- AC-A1.1 채널 추가 화면에 검색 입력창이 있고, 입력 시 후보 채널 목록(제목·핸들·썸네일·구독자수 등 가능한 항목)이 표시된다.
- AC-A1.2 후보를 선택하면 해당 channelId로 구독이 저장된다(기존 subscriptions 저장 로직 재사용, user_id+channel_id UNIQUE).
- AC-A1.3 이미 구독 중인 채널은 목록에서 "구독됨"으로 표시되고 중복 저장되지 않는다.
- AC-A1.4 사칭·동명 채널 대비로 후보를 복수 노출한다(단일 자동선택 금지).

**REQ-A2** 직접 입력 경로를 폴백으로 유지한다.
- AC-A2.1 검색에 원하는 채널이 없거나 API 상한 소진 시, 기존 핸들/URL 직접 입력으로 추가할 수 있는 경로가 남아 있다.

## B. 검색 소스 우선순위 (쿼터 보호 핵심)

**REQ-B1** 검색은 로컬 카탈로그 → 캐시 → API 순으로 해결한다.
- AC-B1.1 먼저 getkkul 로컬 채널 카탈로그(`channel_catalog`)에서 제목 부분일치(like)로 검색한다. 충분한 결과가 있으면 API를 호출하지 않는다.
- AC-B1.2 로컬 결과가 부족하면, 동일 검색어의 최근 캐시(`channel_search_cache`)를 확인한다. 유효 캐시가 있으면 API를 호출하지 않는다.
- AC-B1.3 로컬·캐시로 부족할 때만 YouTube Data API `search.list`(type=channel)를 호출한다.
- AC-B1.4 API 결과는 캐시에 저장하고, 사용자가 실제 선택·저장한 채널은 `channel_catalog`에 축적한다(다음부터 로컬에서 잡힘).

## C. 쿼터 가드 (API 호출 억제)

**REQ-C1** API 호출을 최소화한다.
- AC-C1.1 디바운스: 타이핑이 멈춘 뒤(예: 400ms) 1회만 검색 트리거. 매 키 입력 호출 금지.
- AC-C1.2 최소 글자수: 2글자 이상일 때만 검색한다.
- AC-C1.3 캐시 TTL: API 결과는 일정 시간(예: 24시간) 재사용한다.

**REQ-C2** 검색 API 쿼터를 감지와 분리·상한 관리한다.
- AC-C2.1 검색용 API 일일 유닛 상한(예산)을 둔다. 상한 도달 시 API 검색을 중단하고 로컬·캐시 + 직접 입력만 제공한다(감지 쿼터 침범 금지).
- AC-C2.2 검색 API 사용량을 집계·기록한다(감지와 구분).
- AC-C2.3 (향후 확장) 검색을 별도 API 키/프로젝트로 물리 격리할 수 있도록, 검색 API 키를 감지와 분리해 참조할 수 있게 설계한다.

## D. 데이터 모델

- **channel_catalog**(id PK, channel_id TEXT UNIQUE, title, handle, thumbnail_url, subscriber_hint, source enum['user_selected','api','detected'], created_at, updated_at) — 검색용 로컬 카탈로그. 검색 확정·감지된 채널을 축적.
- **channel_search_cache**(id PK, query_norm TEXT, results jsonb, fetched_at, expires_at, UNIQUE(query_norm)) — 검색어 정규화(소문자·트림) 기준 API 결과 캐시.
- **search_api_usage**(id PK, day DATE UNIQUE, units_used INT default 0, cap INT) — 검색 API 일일 유닛 집계·상한.
- (기존 subscriptions 재사용: 저장은 channel_id 기준.)

RLS: channel_catalog / channel_search_cache / search_api_usage는 공용 읽기(카탈로그·캐시) 또는 서비스 롤(사용량). 구독 저장은 기존 RLS.

## E. 비기능 요구사항

- E1 감지 보호: 검색이 감지 폴백 쿼터를 침범하지 않아야 한다(상한·집계로 강제).
- E2 정규화: 검색어는 소문자·공백 정리 후 캐시 키로 사용(캐시 적중률↑).
- E3 회복력: API 실패·상한 시에도 검색 UI는 로컬/직접입력으로 동작(빈 화면 금지).
- E4 보안: API 키는 env로만. 검색 결과에 사칭 경고가 필요하면 UI에서 복수 노출로 사용자 판단 유도.
- E5 기존 CLAUDE.md·PRD·SSR의 규칙·스택·격리 경계를 따른다.

## F. 수용성 시나리오 & 통과 기준

- S1 로컬 우선: 카탈로그에 있는 인기 채널 검색 → API 호출 0회로 결과 표시.
- S2 캐시 적중: 같은 검색어 반복 → 2번째부터 API 호출 0회.
- S3 API 폴백: 카탈로그·캐시에 없는 신규 채널 → API 1회 호출 후 결과, 이후 캐시/카탈로그 축적.
- S4 디바운스/최소글자: 빠른 연속 타이핑·1글자 입력 시 API 호출이 발생하지 않는다.
- S5 상한 가드: 검색 일일 상한 도달 → API 검색 중단, 로컬+직접입력만 제공, 감지 쿼터 미침범.
- S6 저장·중복: 후보 선택 저장 → subscriptions 1건, 재선택 시 중복 없음.
- 통과 기준: 일반 사용에서 API 호출률(검색 대비) 목표 이하(예: 캐시·카탈로그 적중으로 신규 채널 검색에만 API 사용), 검색으로 인한 감지 쿼터 침범 0건.

## G. 확정값 요약

로컬 카탈로그 우선 + 캐시(TTL 24h) + API 폴백(디바운스 400ms·최소 2글자·일일 상한) + 직접입력 폴백. 검색 확정 채널은 카탈로그 축적. 감지 쿼터 침범 0건이 최우선. 별도 API 키 물리격리는 향후 확장.
