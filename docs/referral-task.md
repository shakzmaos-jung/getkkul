# 작업 지시 — getkkul 친구추천 크레딧 시스템

**대상 에이전트** Claude Code · **상태** 실행 준비 완료

---

## 목표

겟꿀에 친구추천 크레딧 프로그램을 구현한다. 사용자가 추천 링크를 공유하고, 친구가 활성화(채널 3개 구독 + 요약 항목 10개 수신)하면 추천인·피추천인 각각에게 크레딧 2,000원을 지급한다. 크레딧은 지급 건별 5년 유효, 만료 임박 순 사용, 예산·상한·어뷰징 가드로 보호된다.

## 단일 진실 공급원

- `docs/referral-spec.md` — 이 기능의 REQ/AC(수용 기준). 모든 결정이 여기 확정돼 있다.
- `docs/referral-prd.md` — 이 프로그램의 목적·정책·예산 모델.
- `CLAUDE.md`, `docs/PRD.md`, `docs/SSR.md` — 프로젝트 전반 규칙·스택·격리 경계.

## 작업 방식 (중요)

- 이 작업은 스키마 변경을 포함하지만, 관련 결정이 스펙에 이미 승인돼 있다. 스펙 범위 안에서는 추가 승인 없이 끝까지 자율로 완성한다(TDD 준수).
- 시작 전 먼저, 이 스펙을 구현하며 내가 결정하거나 사전에 해줘야 할 것이 있으면 하나의 목록으로 모아 물어보고 답을 기다린다(없다면 없다고 말하고 바로 진행). 이후에는 스펙에 없는 새로운 에스컬레이션 사유가 생길 때만 멈춘다.

## 확정된 결정 (스펙에 반영됨 — 재확인 불필요)

- 보상: 추천인 2,000원 + 피추천인 2,000원, 활성화 시 지급.
- 활성화: 채널 구독 3개 AND 요약 항목 10개 수신(발송 횟수 아님, 실제 콘텐츠 건수).
- 상한: 1인 50,000원. 예산 킬스위치: 누적 발행 5,000,000원 도달 시 신규 지급 중단.
- 크레딧: 지급 건별 로트 저장, 유효기간 5년, 사용 시 만료 임박 순(FIFO) 차감, 결제액 50%까지.
- 화면: 크레딧 원장(잔액·만료임박·내역) + 추천 현황(피추천인 진행률·상태만, 상세 비노출). 설정 메뉴 내.
- 알림: 지급 시 기존 notify 레이어로 이메일(+푸시).
- 어뷰징 방지: 정규화 이메일 해시 + 보상 이력 플래그(탈퇴 후에도 보존), 기기·결제수단 지문, 자기추천 차단.
- 탈퇴: 본인 크레딧 즉시 소멸, 추천인 보상은 유지.

## 구현 범위

`docs/referral-spec.md`의 REQ A~L을 모두 구현한다. 요약: 추천 코드·링크(A), 가입 연결(B), 활성화 판정(C), 양방향 지급(D), 예산·상한 가드(E), 지급 건별 원장·FIFO 사용(F), 크레딧·추천현황 화면(G), 지급 알림(H), 어뷰징 방지(I), 탈퇴 처리(J), 데이터 모델(K).

### 범위 경계 (지금 하지 말 것)
- 실제 유료 결제 연동은 유료 티어가 아직 없으므로 하지 않는다. 크레딧 사용 로직(FIFO·50% 상한)은 **함수로 구현·테스트만** 하고, 실제 체크아웃에는 연결하지 않는다(REQ-F2.3). 유료화 시점(v2)에 연결.

## 품질 기준 (TDD)

- 각 AC를 실패 테스트로 먼저 작성한 뒤 구현한다.
- 특히 다음은 반드시 테스트 우선:
  - "활성화 판정(채널3+요약10, 1회 전이)"
  - "예산 킬스위치·1인 상한 가드(경합 시 초과 방지, 트랜잭션 원자성)"
  - "FIFO 만료 임박 순 차감 + 결제 50% 상한"
  - "재가입 어뷰징 차단(정규화 이메일 해시·보상 이력)"
  - "탈퇴 시 본인 크레딧 소멸 + 추천인 보상 유지"
- 금액 관련 모든 지급/차감/만료/소멸은 DB 트랜잭션으로 원자적으로 처리한다.
- 어뷰징 방지용 최소 식별정보 외에는 개인정보 최소수집을 지킨다.

## 문서 정합성

- 구현하며 스펙과 코드가 어긋나면 `docs/referral-spec.md`를 실제 구현에 맞춰 갱신하고, 어떤 결정이 왜 바뀌었는지 `docs/adr/`에 기록한다.
