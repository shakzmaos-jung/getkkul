# 작업 지시 — getkkul 파이프라인 신뢰성 개선 (누락·지연 근본 해결)

**대상 에이전트** Claude Code · **상태** 실행 준비 완료

---

## 목표

유튜브 콘텐츠의 누락·지연을 근본적으로 해결한다. 감지를 폴링 → WebSub 푸시로 전환하고, 전사 실패를 회복 가능하게 만들며, 관리형 트랜스크립트 API로 획득을 안정화하고, 처리를 분리·관측 가능하게 한다.

## 단일 진실 공급원

- `docs/pipeline-reliability-spec.md` — 이 개선의 REQ/AC. 모든 요구사항이 여기 있다.
- `CLAUDE.md`, `docs/PRD.md`, `docs/SSR.md` — 프로젝트 전반 규칙·스택·격리 경계.
- 현재 관련 코드: `lib/pipeline/detect.ts`, `acquire.ts`, `fetch-content.ts`, `youtube-content.ts`, `health.ts`, `.github/workflows/pipeline.yml`, `app/api/*`.

## 작업 방식 (중요)

- 이 작업은 스키마 변경·새 외부 의존성(관리형 API, WebSub)·시크릿을 포함한다. CLAUDE.md 에스컬레이션 항목이다.
- 따라서 **먼저 구현을 시작하지 말고**, 아래 "사전 결정/작업"(spec K절)을 하나의 목록으로 모아 나에게 제시하고 답을 기다린다.
- 내가 답한 뒤에는, spec 범위 안에서 추가 승인 없이 단계 순서대로 끝까지 자율 완성한다(TDD 준수). spec에 없는 새 에스컬레이션 사유가 생길 때만 멈춘다.

## 사전 결정/작업으로 나에게 물어볼 것 (spec K절)

1. 관리형 트랜스크립트 API 선택(Supadata vs TranscriptAPI) — 장단점 요약해 추천하고 내 선택을 받는다. 선택 후 나는 API 키를 시크릿에 등록한다.
2. 획득 순서(관리형 API 주경로+yt-dlp 백업 vs 반대) — 추천 후 확정.
3. WebSub 콜백 시크릿(verify_token, HMAC secret) — 생성해 값을 출력하고, 내가 시크릿에 등록한다. 콜백 URL은 https://getkkul.vercel.app/api/webhooks/youtube.
4. 처리 워커 트리거 방식(외부 크론 dispatch vs Supabase pg_cron/Edge Functions) — 추천 후 확정.
5. 외부 하트비트 모니터 선택 — 추천 후 확정(내가 계정/설정).

## 구현 범위 & 순서 (spec J절 우선순위대로)

- **1단계(급한 불)**: B(failed 재시도화) + C(관리형 API 폴백). 먼저 배포해 누락을 즉시 줄인다.
- **2단계(근본)**: A(WebSub 푸시 감지 + 재구독 + 폴링은 백업 유지) + E(창 확대·백필).
- **3단계(운영)**: D(감지·처리 잡 분리, 트리거 이전) + F(pipeline_runs 관측·하트비트) + G(웹훅 즉시처리·자가치유 보정).

각 단계는 독립적으로 배포 가능하게 만들어, 1단계만으로도 개선이 체감되게 한다.

## 품질 기준 (TDD)

- 각 AC를 실패 테스트로 먼저 작성한 뒤 구현한다.
- 특히 다음은 반드시 테스트 우선:
  - "WebSub GET 챌린지 검증·에코, POST HMAC 서명 검증, Atom 파싱→upsert 멱등"
  - "failed 재시도: next_retry_at 도래 시 재획득, 영구/일시 실패 구분, 최대 재시도 후 종점"
  - "획득 티어 폴백 순서(관리형 API↔yt-dlp)"
  - "백필: 채널 격차 감지 시 pageToken 페이지네이션"
  - "웹훅 즉시처리 실패가 재시도 큐로 안전하게 떨어짐(유실 없음)"
- 시크릿은 코드에 넣지 말고 환경변수로만 참조. 콜백은 HMAC 검증.
- 개별 영상/채널 실패가 전체 잡을 중단시키지 않는다.

## 수용성 검증 (반드시 이 기준으로 판정)

spec L절의 정량 기준과 시나리오(S1~S8)를 검증 대상으로 삼는다. 두 층으로 구현한다.

- **자동화 계약 테스트(CI 상시)**: S1~S7의 로직을 모의 WebSub 페이로드·강제 실패 주입·백필 픽스처로 판정. 외부 게시에 의존하지 않는 부분은 전부 자동 테스트로 커버한다.
- **종단 수용 체크리스트(배포 후 사람 1회)**: 실제 테스트 채널로 S1~S4를 실측하고, 정량 기준(감지 p95 ≤ 5분, 요약 p95 자막 15분·비자막 30분, 누락률 0%, 종점 failed율 ≤ 2%)의 실측값을 완료 보고에 제출한다. S8(1주 집계)은 pipeline_runs 지표로 확인하는 절차를 함께 제시한다.

"코드가 돈다"가 아니라 "위 정량 기준을 실측으로 충족한다"가 완료 판정 기준이다.

## 문서 정합성

- 감지 방식이 폴링→푸시로 바뀌므로, `docs/SSR.md`의 감지 관련 AC(REQ-C1 계열)를 WebSub 우선 + 폴링 백업 + 실패 재시도 흐름에 맞춰 갱신한다.
- 변경된 결정은 `docs/adr/`에 기록한다.
