# 작업 지시 — getkkul 채널 검색 등록

**대상 에이전트** Claude Code · **상태** 실행 준비 완료

---

## 목표

getkkul 안에서 유튜브 채널을 제목으로 검색(like)해 후보 목록에서 선택·저장하는 UX를 추가한다. 사용자가 유튜브 앱을 오가며 핸들·URL을 찾는 마찰을 없애 구독 등록(락인)을 늘린다.

## 단일 진실 공급원

- `docs/channel-search-spec.md` — 이 기능의 REQ/AC. 모든 결정이 여기 있다.
- `CLAUDE.md`, `docs/PRD.md`, `docs/SSR.md` — 프로젝트 전반 규칙·스택·격리 경계.
- 관련 코드: 기존 subscriptions 저장 로직, `lib/youtube`(Data API 사용부), 채널 추가 화면(components/subscriptions·app/subscriptions).

## 작업 방식 (중요)

- 스키마 변경을 포함하지만 결정은 스펙에 승인돼 있다. 스펙 범위 안에서 추가 승인 없이 끝까지 자율 완성한다(TDD 준수).
- 시작 전, 사전에 내가 정하거나 해줘야 할 것이 있으면 하나로 모아 물어보고(없으면 없다고 말하고 진행), 그 외에는 스펙 밖 에스컬레이션 사유가 생길 때만 멈춘다.

## 확정된 결정 (스펙 반영 — 재확인 불필요)

- 검색 우선순위: 로컬 카탈로그(`channel_catalog`) → 캐시(`channel_search_cache`, TTL 24h) → YouTube Data API `search.list`(type=channel) → 직접 입력(URL/핸들) 폴백.
- 쿼터 가드: 디바운스 400ms, 최소 2글자, 검색 API 일일 유닛 상한(`search_api_usage`)으로 감지 쿼터 침범 금지.
- 검색으로 확정·저장된 채널은 `channel_catalog`에 축적해 다음부터 무료화.
- 사칭·동명 대비 후보 복수 노출(단일 자동선택 금지). 이미 구독 채널은 "구독됨" 표시.
- 별도 API 키 물리 격리는 향후 확장(지금은 상한·집계로 논리 분리).

## 구현 범위

spec REQ A~E 전부: 검색 UI(디바운스·최소 글자수·후보 목록·선택 저장·구독됨 표시), 소스 우선순위 로직(로컬→캐시→API), 쿼터 가드(상한·집계·감지 보호), 카탈로그 축적, 직접입력 폴백 유지, 데이터 모델(channel_catalog, channel_search_cache, search_api_usage).

## 품질 기준 (TDD)

- 각 AC를 실패 테스트로 먼저 작성한 뒤 구현한다.
- 특히 반드시 테스트 우선:
  - "소스 우선순위: 로컬/캐시 적중 시 API 미호출"
  - "디바운스·최소 글자수로 API 호출 억제"
  - "검색 일일 상한 도달 시 API 중단 + 로컬/직접입력 유지 + 감지 쿼터 미침범"
  - "검색 확정 채널의 카탈로그 축적 → 다음 검색 로컬 적중"
  - "구독 저장 멱등(중복 없음)"
- API 키는 env로만 참조. 검색 API 사용량은 감지와 구분해 집계.

## 수용성 검증

- spec F절 시나리오 S1~S6과 통과 기준(검색으로 인한 감지 쿼터 침범 0건, 로컬·캐시 적중으로 신규 채널 검색에만 API 사용)을 검증 대상으로 삼는다.
- API 미호출을 검증하는 계약 테스트(모의)와, 실제 검색 몇 건으로 쿼터 소비를 확인하는 체크리스트를 함께 제시한다.

## 문서 정합성

- 채널 등록 방식이 확장되므로, 필요 시 `docs/SSR.md`의 소스 등록 관련 AC(REQ-B1 계열)를 검색+직접입력 병행으로 갱신한다. 변경은 `docs/adr/`에 기록.
