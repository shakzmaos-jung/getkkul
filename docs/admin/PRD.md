# PRD — getkkul 관제 어드민 (Control Tower Admin)

> **성격**: getkkul 관제 어드민의 제품 요구사항 정의(무엇/왜). 검증 가능한 요구(REQ)와 수용 기준(AC)은 `docs/admin/SSR.md`, 마일스톤 실행은 `docs/admin/EXECUTION-PLAN.md`.
> **상속**: 루트 `CLAUDE.md`, `docs/PRD.md`, `docs/SSR.md`, `docs/adr/*`. 충돌 시 상위 컨스티튜션 우선.
> **버전**: v1.0 · **대상 리포**: `shakzmaos-jung/getkkul` (v0.6.0 기준)

## 1. 개요
getkkul 서비스를 **읽어서 관제**하는 별도 어드민 웹. 인프라 헬스·다이제스트 파이프라인·그로스·비용/쿼터·보안·알림을 한 화면에서 파악하는 "믿고 맡길 데브옵스 팀 같은 관제탑". 어드민은 **기존 스키마의 read-layer**이며, 신규 DB 객체는 최소·additive.

## 2. 배경 / 왜 지금
- 루트 `CLAUDE.md`의 "범위 경계"에서 **운영자 어드민 대시보드는 v2/v3 항목으로 명시**돼 있었다. 서비스가 v0.6.0(멤버십·레퍼럴·4슬롯 발송·피드백 루프)까지 성숙해, 관제 필요가 실체화됨.
- 비개발자(운영자) 1인 체계에서 "인프라가 도는지·파이프라인이 밀리는지·비용이 새는지·보안 구멍은 없는지"를 사용자보다 먼저 감지해야 한다.

## 3. 사용자
- **Master**: 운영자(Chess) 1인. 전체 read + 초대/취소 + 수동 실행 권한.
- **Sub-master**: master가 초대한 신뢰 그룹 소수. 제한 권한(주로 조회).
- 공개 아님. 셀프가입 없음.

## 4. 목표 / 비목표
**목표**
- 서비스 상태를 30초 내 파악(관제 홈, KPI ≤6).
- 파이프라인(감지→수집→요약→발송) 지연·실패를 조기 감지.
- LLM/YouTube/이메일 비용·쿼터를 사용자별·기능별로 귀속·모니터.
- 성장(구독자·활성화·리텐션·레퍼럴)을 한 곳에서.
- 보안 상태(시크릿·의존성·헤더·IDOR·이상탐지)를 상시 가시화.

**비목표(이번 범위 아님)**
- 사용자 웹 기능/파이프라인 로직 변경(어드민은 관측·수동 트리거만).
- 결제/과금 로직 변경. 외부 APM(Datadog 등) 도입.
- 세만틱 중복 제거, 신규 콘텐츠 소스 등 v2/v3 서비스 기능.

## 5. 정보구조 (8개 모듈)
관제(monitoring) 7 + 운영(ops) 1.

| 모듈 | 목적 |
|------|------|
| 관제 홈 (Overview) | 서비스 신호등 + 오늘 배치 결과 + KPI 6종 |

| 다이제스트 파이프라인 (Pipeline) | 4단계 배치 타임라인·채널별 처리·재시도 큐 |
| 그로스 (Growth) | 구독자·활성화·리텐션·퍼널·레퍼럴 |

## 6. 디자인
- **출시 테마 = Linear**(다크 캔버스 + 4단 서피스 사다리 + 라벤더 프라이머리 + 헤어라인, 그림자 대신 서피스 엘리베이션). 폰트 Inter + Geist Mono.
- **멀티테마 기능은 아키텍처로 내장**(디자인 토큰 레이어 + 테마 레지스트리)하되, 출시는 `linear` 단일 테마. 이후 테마 추가는 토큰 블록 추가만으로.

## 7. 성공 기준
- 관제 홈에서 서비스 이상(배치 실패율↑, 요약 백로그↑, 인시던트)을 즉시 식별 가능.
- 모든 지표가 실제 데이터(RPC) 기반(하드코딩·목 없음).
- 기존 사용자 웹 회귀 0(270+ 테스트 green 유지).
- 어드민 접근이 서버사이드에서 이중 검증(비-admin 차단, IDOR 방지).

## 8. 범위 & 마일스톤 (요약)
M0 모노레포 전환 → M1 셸+Linear 토큰+인증 → M2 관제 홈 → M3 파이프라인 → M4 비용·쿼터 → M5 그로스 → M6 보안 → M7 알림 → M8 운영 데이터. 상세는 `EXECUTION-PLAN.md`.

## 9. 리스크 / 에스컬레이션
- M0 모노레포 전환의 회귀 위험(불변 조건: 기존 테스트 green).
- 신규 DB 객체(`admin_users`·관제 RPC)·신규 의존성(차트 라이브러리)·파괴적 액션(수동 실행)은 **진행 전 정지·승인**(SSR §보안·EXECUTION-PLAN §9).

## 10. 원칙 증보 (v1.1): Living Spec & Sync 거버넌스
- 이 SOT(PRD/SSR/EXECUTION-PLAN)는 아카이브가 아니라 **코드와 함께 사는 living spec**이다. 구현 결정이 스펙과 갈리면(드리프트) 이를 감지·분류·기록해 동기화한다.
- **목표 추가**: SOT↔코드 드리프트를 spec-sync(서브에이전트 + Stop hook)로 자동 관리하고, 그 내역을 `docs/admin/SYNC-LOG.md`에 감사 추적으로 남긴다(EXECUTION-PLAN §10).
- 이는 "PRD→SSR→SDD→TDD→배포를 한 큐로 0→1" 실험에서 **문서와 구현의 정합성을 측정 가능하게** 하는 핵심 장치다. 원칙: 스펙에도 코드와 동일한 거버넌스(버전관리·리뷰·CI 게이트·기록)를 적용한다.
